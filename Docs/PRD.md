# FreeShot 产品需求文档（PRD）

---

## 1. 文档说明

本文档用于描述 FreeShot 当前阶段的**产品目标、功能范围、真实进度、关键风险、验收标准与发布标准**。

这不是早期立项草案，而是一份面向当前版本的**执行型 PRD**：

- 要能反映当前项目真实状态
- 要能指导接下来的开发优先级
- 要能约束对外宣传口径
- 要能作为测试与发布判断依据

---

## 2. 产品概述

| 项目 | 内容 |
|------|------|
| **产品名称** | FreeShot |
| **产品定位** | 免费 macOS 截图 / 录屏工具，面向内容创作、演示、教学与开发者场景 |
| **目标平台** | macOS 13.0+ |
| **核心价值** | 免费、开源、本地优先，提供接近商业截图录屏工具的核心体验 |
| **开源协议** | MIT |
| **目标用户** | 自媒体创作者、在线教育者、开发者演示、产品演示人员 |

### 2.1 产品愿景

FreeShot 的目标不是做一个“功能看起来很多、但真实使用容易翻车”的演示项目，而是做一个**用户可以在真实 macOS 环境下完成截图、录屏、导出与轻量后处理主流程**的工具。

现阶段的产品目标非常明确：

> **先把截图 / 录屏主链路做稳，再谈扩展与外围能力。**

### 2.2 目标用户画像

#### A. 自媒体创作者
需求：快速截图、录屏、摄像头叠加、简单剪辑、导出即用。  
关注点：稳定、快、操作简单、免费。

#### B. 开发者 / 产品经理
需求：录制产品演示、Bug 复现、功能说明、交互演示。  
关注点：区域录屏准确、键盘鼠标展示、历史管理、快速导出。

#### C. 在线教育 / 知识分享用户
需求：录教学视频、带摄像头讲解、展示键盘操作。  
关注点：录制稳定、摄像头画中画、倒计时、麦克风可用。

## 3. 市场与竞争定位

### 3.1 对标产品

- CleanShot X
- Shottr
- Kap
- Loom
- Screen Studio

### 3.2 用户为什么不用现成产品，要看 FreeShot？

FreeShot 的意义不在于“比所有商业产品功能都多”，而在于提供以下组合价值：

1. **免费可用**：核心能力不依赖订阅才能使用
2. **开源透明**：实现、问题、方向可见，不是黑盒产品
3. **本地优先**：核心截图与录屏主流程不依赖云服务
4. **创作者友好**：围绕录屏、摄像头、键盘、鼠标、简单导出做整合
5. **中文语境友好**：更适合本地创作者、教程作者与开发演示场景

### 3.3 当前阶段不应盲目对标的方向

FreeShot 当前阶段不应追求与 Screen Studio 之类高完成度商业产品在“视觉包装”和“高级编辑”上硬拼，而应优先守住：

- 录制成功率
- 坐标准确性
- 权限体验
- 导出结果稳定性
- 连续使用不崩

## 6. 当前阶段必须解决的问题

### 6.1 P0 问题（发布前必须解决）

1. **录屏黑屏 / 空文件 / 写入异常**
   - 直接决定录屏主功能是否成立
   - 必须稳定处理首帧时间戳与写入 session

2. **区域录屏坐标映射错误**
   - AppKit / CoreGraphics 坐标系转换错误会导致区域偏移
   - 在多显示器与 Retina 环境下风险更高

3. **暂停逻辑与实际导出结果不一致**
   - 不能只暂停 UI，必须真正控制帧写入链路

4. **重复录制崩溃 / 状态清理不彻底**
   - 一次录完再录崩溃，会直接摧毁可用性

5. **录屏文件生成稳定性不足**
   - 包括时长异常、输出失败、无法结束等问题

### 6.2 P1 问题（发布前尽量解决）

1. 多显示器下区域映射与选区结果不稳定
2. Retina 缩放导致的坐标偏移
3. 权限失败提示不够明确
4. 连续多次录制稳定性不足
5. 摄像头叠加在不同分辨率下的表现一致性不够

### 6.3 P2 问题（可延后）

1. 系统声音录制方案完善
2. 云上传能力落地
3. 滚动截图成熟化
4. Touch Bar 等长尾特性的持续维护

## 7. 用户主流程

### 7.1 截图主流程

```text
状态栏图标 / 快捷键 → 选择截图模式（区域 / 窗口 / 全屏）→ 完成截图 → 预览 / 标注 / 保存 / 进入历史
```

### 7.2 录屏主流程

```text
状态栏图标 / 快捷键 → 选择录屏模式（区域 / 全屏）→ 可选倒计时 / 摄像头 / 键盘 / 鼠标 → 开始录制 → 暂停 / 继续 / 停止 → 输出 MP4 → 可选裁剪 / GIF 导出
```

### 7.3 当前阶段最低可接受体验

用户至少必须稳定完成：

1. 发起截图
2. 发起录屏
3. 正常停止录屏
4. 获取可播放且内容正确的文件
5. 连续多次使用不明显崩溃

如果以上路径不能稳定达成，则不能视为产品已进入可发布阶段。

## 8. 技术方案

### 8.1 技术栈

| 层级 | 技术 |
|------|------|
| **UI** | SwiftUI + AppKit |
| **屏幕采集** | ScreenCaptureKit |
| **摄像头** | AVFoundation |

| **OCR** | Vision / 本地识别链路 |

### 8.2 当前技术重点

当前重点不再是单纯加功能，而是：

- 统一 AppKit / CG 坐标系统
- 稳定首帧时间戳与 AssetWriter session
- 理顺录制状态机与会话清理逻辑
- 提升多显示器 / Retina 场景稳定性
- 建立真实可重复的真机验证机制

## 10. 验收标准（量化版）

### 10.1 截图验收标准

- 区域 / 窗口 / 全屏截图主流程可连续执行 **10 次以上无崩溃**
- 生成截图内容与用户选择区域一致，无明显偏移
- 预览、保存、历史至少在真机上完成一次完整验证
- 标注后保存结果正确，不出现明显丢失或空白

### 10.2 录屏验收标准

- 区域录屏与全屏录屏都能在真机上完成 **10 次连续录制**，无主流程崩溃
- 录制停止后生成的 MP4 可正常播放
- 不允许高频出现以下问题：
  - 黑屏
  - 空文件
  - 时长异常
  - 停止后无法生成文件
- 暂停 / 继续逻辑必须真实作用于录制结果，不能只停 UI 不停写入
- 连续重新录制时不能因为状态未清理而崩溃

### 10.3 多显示器 / Retina 验收标准

- 在至少 2 组不同显示环境下完成验证：
  1. 单屏 Retina
  2. 双屏 / 外接显示器
- 用户框选区域与录制 / 截图结果应保持一致
- 不允许出现明显坐标翻转、整体偏移、尺寸错误

### 10.4 权限体验验收标准

- 首次录屏失败时，必须给出明确权限提示
- 当缺少屏幕录制权限、麦克风权限时，用户能明确知道下一步去哪里开启
- 权限未开时不能无提示失败

## 11. 发布标准

### 11.1 什么叫“开发版可演示”
满足以下条件即可：

- 截图主流程基本可用
- 录屏主流程在开发者机器上可跑通
- 已知问题被明确记录
- README 与 PRD 不夸大能力

### 11.2 什么叫“可对外公开测试”
必须满足：

- 区域 / 全屏录屏主流程真机验证通过
- 黑屏、暂停、重复录制崩溃等 P0 问题大幅收敛
- 权限失败路径可解释
- README 中对“开发中 / 待验证”能力表述准确

### 11.3 什么叫“可稳定发布”
必须满足：

- 截图 / 录屏主链路在真实设备上有稳定验证结果
- 连续录制、暂停、导出、重复录制都无高频重大故障
- 多显示器 / Retina 至少具备基础可用性
- README、测试清单、PRD 三者一致
- 不再依赖“功能已经有了，但还没验证”的表述掩盖稳定性问题

## 12. 当前对外宣传边界

以下内容在完全验证前，不应作为“稳定成熟能力”对外宣传：

- 滚动截图
- 多显示器完整支持
- 系统声音录制
- 云上传
- 长时间连续录制稳定性

可以写：

- 已开发
- 开发中
- 待验证

但不能写成“稳定可用”或“已完整支持”。

## 13. 接下来两周的执行重点

当前版本不应继续大范围加功能，接下来两周只建议聚焦以下三件事：

### 重点 1：录屏主链路稳定性收敛

目标：
- 修掉黑屏、暂停、坐标、重复录制崩溃
- 把录屏从“能跑”推进到“基本稳”

### 重点 2：真机测试结果沉淀

目标：
- 让测试清单不是摆设
- 每个关键场景至少有真机验证结果
- 明确哪些问题已经解决，哪些还没解决

### 重点 3：统一文档口径

目标：
- README
- PRD
- TEST_CHECKLIST
- 开发计划

这几份文档必须反映同一个现实，不要一份写“已完成”，另一份还写“待开发”。

## 14. 开源与产品原则

FreeShot 当前阶段坚持以下原则：

- **免费可用**：优先保障核心主流程免费可用
- **开源透明**：文档与代码状态尽量一致，不搞虚假完成度
- **本地优先**：核心体验不依赖云服务才能成立
- **稳定优先**：先把主链路做稳，再打磨花活
- **少吹牛**：未验证就是未验证，不假装已经成熟

## 15. 结论

FreeShot 当前已经不是“从 0 到 1”的概念项目，而是一个**功能已大体成型、正在冲击稳定发布门槛**的 macOS 工具项目。

现阶段最重要的目标不是继续加功能，而是：

> **把截图 / 录屏主链路真正打磨到可以稳定交付。**

只要主链路稳住，FreeShot 才有资格从“开源项目”走向“真正能长期使用的产品”。
